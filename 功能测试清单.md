## 部署前准备

### 1. Railway 环境变量更新
在 Railway → 你的服务 → Variables 中添加/修改以下变量：

| 变量名 | 值 | 说明 |
|--------|-----|------|
| `DAILY_REQUEST_LIMIT` | `100` | 全局每日上限（防滥用） |
| `FREE_DAILY_LIMIT` | `2` | 免费用户每日次数 |
| `RATE_LIMIT_PER_MINUTE` | `5` | 速率限制（已有） |
| `ADMIN_KEY` | `gaokao-admin-2026-secret` | 管理员密钥（自己定一个） |

**注意**：原有 LLM 变量（LLM_API_BASE_URL, LLM_API_KEY, LLM_MODEL）保持不变。

```bash
  -H "X-Admin-Key: gaokao-admin-2026-secret" \
  -H "Content-Type: application/json" \
  -d '{"count": 10, "credits_per_code": 1}'

  -H "X-Admin-Key: gaokao-admin-2026-secret"
```

## ⚠️ 已知限制

1. **数据持久化**：兑换码数据存储在 `data/redemption_codes.json`。Railway默认重启后会丢失。解决方案：
   - MVP阶段可接受（一年只卖一个月，重启概率低）
   - 如需持久化，Railway支持Volume挂载（$5计划内免费）
   - 或迁移到Railway PostgreSQL（更稳定）

2. **付费次数验证**：当前方案前端localStorage记录付费次数，服务端只做全局上限。如果用户清空localStorage，付费次数会丢失（但兑换码已标记为used，不能重复兑换）。MVP阶段可接受。

3. **单人管理**：所有兑换码由你一个人通过curl命令生成和管理，没有后台界面。够用。
